# Progress Writing Rules

`progress.md` is a concise handoff log, not a transcript.

## Purpose

Use it to record:

- what changed
- why it changed
- which durable files were updated
- what the recommended next step is

## Append-Only Rule

- append new sections
- do not rewrite history unless the user explicitly asks

## Good Entry Shape

Each entry should ideally cover:

- date or timestamp
- action taken
- relevant files changed
- concise reason or context
- recommended next step

## Keep It Short

Do not put these into `progress.md`:

- long brainstorming transcripts
- repeated large summaries from `plan.md`
- duplicated knowledge content from `project_knowledge`

## Role-Specific Guidance

- init: record initialization outputs and next recommended feature
- developer: record one-task cycle progress and any durable state updates
- reviewer: record only stable review conclusions worth handing off
- planner: record planning or backlog changes and why they matter
